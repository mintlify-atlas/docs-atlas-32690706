The sample plugin repository is intentionally minimal. Every file has a specific role in the build pipeline, plugin metadata, or runtime behavior.

## File tree

```
obsidian-sample-plugin/
├── src/
│   ├── main.ts              # Plugin entry point
│   └── settings.ts          # Settings interface and settings tab
├── .github/
│   └── workflows/
│       └── lint.yml         # Automated lint on every push
├── main.js                  # Compiled output (generated, do not edit)
├── manifest.json            # Plugin metadata required by Obsidian
├── styles.css               # Optional CSS bundled with the plugin
├── versions.json            # Plugin version → minimum app version map
├── version-bump.mjs         # Version automation script
├── esbuild.config.mjs       # Build configuration
├── tsconfig.json            # TypeScript compiler options
├── eslint.config.mts        # ESLint configuration
├── package.json             # npm scripts and devDependencies
└── package-lock.json        # Locked dependency tree
```

<Tip>
  Do not commit `main.js` or `node_modules/` to version control. The sample repo's `.gitignore` already excludes them. Committing `main.js` creates merge conflicts and bloats your history — release it only as a GitHub release asset.
</Tip>

## Source files

### `src/main.ts` — plugin entry point

This is the file Obsidian loads. It exports a single default class that extends `Plugin` and implements two lifecycle methods:

- **`onload()`** — runs when the plugin is enabled. Register all commands, ribbon icons, status bar items, settings tabs, and event listeners here.
- **`onunload()`** — runs when the plugin is disabled. Listeners registered with `registerDomEvent`, `registerEvent`, and `registerInterval` are cleaned up automatically; add any additional teardown logic here.

```typescript
export default class MyPlugin extends Plugin {
  settings: MyPluginSettings;

  async onload() {
    await this.loadSettings();

    this.addRibbonIcon('dice', 'Sample', (evt: MouseEvent) => {
      new Notice('This is a notice!');
    });

    this.addCommand({
      id: 'open-modal-simple',
      name: 'Open modal (simple)',
      callback: () => {
        new SampleModal(this.app).open();
      }
    });

    this.addSettingTab(new SampleSettingTab(this.app, this));
  }

  async loadSettings() {
    this.settings = Object.assign({}, DEFAULT_SETTINGS, await this.loadData() as Partial<MyPluginSettings>);
  }

  async saveSettings() {
    await this.saveData(this.settings);
  }
}
```

Keep `main.ts` focused on lifecycle and registration. Move feature logic into separate modules under `src/`.

### `src/settings.ts` — settings interface and settings tab

Defines the shape of your plugin's configuration, the default values, and the `PluginSettingTab` subclass that renders the settings UI.

```typescript
export interface MyPluginSettings {
  mySetting: string;
}

export const DEFAULT_SETTINGS: MyPluginSettings = {
  mySetting: 'default'
}

export class SampleSettingTab extends PluginSettingTab {
  plugin: MyPlugin;

  display(): void {
    const {containerEl} = this;
    containerEl.empty();

    new Setting(containerEl)
      .setName('Settings #1')
      .setDesc('It\'s a secret')
      .addText(text => text
        .setPlaceholder('Enter your secret')
        .setValue(this.plugin.settings.mySetting)
        .onChange(async (value) => {
          this.plugin.settings.mySetting = value;
          await this.plugin.saveSettings();
        }));
  }
}
```

## Configuration files

### `manifest.json` — plugin metadata

Obsidian reads this file to identify, display, and validate your plugin. Every field here is checked against the community catalog schema when you submit.

```json
{
  "id": "sample-plugin",
  "name": "Sample Plugin",
  "version": "1.0.0",
  "minAppVersion": "0.15.0",
  "description": "Demonstrates some of the capabilities of the Obsidian API.",
  "author": "Obsidian",
  "authorUrl": "https://obsidian.md",
  "fundingUrl": "https://obsidian.md/pricing",
  "isDesktopOnly": false
}
```

| Field | Required | Notes |
|---|---|---|
| `id` | Yes | Stable identifier. Must match the plugin folder name. Never rename after release. |
| `name` | Yes | Display name in the community plugin browser. |
| `version` | Yes | SemVer `x.y.z`. Must match the GitHub release tag exactly. |
| `minAppVersion` | Yes | Keep up to date when you use newer APIs. |
| `description` | Yes | One-sentence summary. |
| `isDesktopOnly` | Yes | `false` unless you depend on Node or Electron APIs. |
| `author` | No | Your name or organization. |
| `authorUrl` | No | Link to your website or profile. |
| `fundingUrl` | No | Single URL string, or an object mapping labels to URLs. |

### `versions.json` — compatibility mapping

Maps each plugin version to the minimum Obsidian app version that supports it. Obsidian uses this to serve the correct plugin version to users on older app releases.

```json
{
  "1.0.0": "0.15.0"
}
```

Add a new entry every time you release. The `npm run version` script updates both `manifest.json` and `versions.json` automatically.

### `esbuild.config.mjs` — build configuration

Bundles `src/main.ts` into a single `main.js` file using [esbuild](https://esbuild.github.io/). The `obsidian` package and all CodeMirror packages are marked as external so they are not included in the bundle — Obsidian provides them at runtime.

Running the script without arguments starts watch mode (`npm run dev`). Passing `production` as an argument produces a minified build without inline source maps (`npm run build`).

### `tsconfig.json` — TypeScript config

Compiles source files in `src/` with strict type checks enabled. The `baseUrl` is set to `src/`, so imports within `src/` resolve relative to that directory.

Notable flags:
- `"noImplicitAny": true` — every value must have an explicit or inferred type.
- `"strictNullChecks": true` — `null` and `undefined` are not assignable to other types without a check.
- `"noImplicitReturns": true` — all code paths in a function must return a value.
- `"noUncheckedIndexedAccess": true` — array and object index access returns `T | undefined`.
- `"isolatedModules": true` — required for esbuild compatibility.
- `"target": "ES6"` — TypeScript's output target (esbuild further transpiles to ES2018 for the bundle).

### `eslint.config.mts` — linting config

Uses `typescript-eslint` and the [`eslint-plugin-obsidianmd`](https://github.com/obsidianmd/eslint-plugin) package, which adds Obsidian-specific rules. Run linting with:

```bash
npm run lint
```

### `package.json` — npm scripts and dependencies

| Script | Command | What it does |
|---|---|---|
| `dev` | `node esbuild.config.mjs` | Compile in watch mode |
| `build` | `tsc -noEmit -skipLibCheck && node esbuild.config.mjs production` | Type-check then produce a minified build |
| `version` | `node version-bump.mjs && git add manifest.json versions.json` | Bump version in `manifest.json` and `versions.json` |
| `lint` | `eslint .` | Run ESLint across the project |

### `version-bump.mjs` — version automation

Reads the current version from `npm_package_version` (set by `npm version`) and writes it back into `manifest.json`. It also adds the new version entry to `versions.json` if it is not already present.

Run a version bump with:

```bash
npm version patch   # 1.0.0 → 1.0.1
npm version minor   # 1.0.0 → 1.1.0
npm version major   # 1.0.0 → 2.0.0
```

### `styles.css` — optional CSS

Obsidian bundles this file with your plugin and makes the styles available when the plugin is enabled. The sample file is a placeholder — delete it if your plugin has no custom styles.

## Recommended src/ structure for larger plugins

As your plugin grows, keep `main.ts` minimal and split feature logic into focused modules:

```
src/
├── main.ts           # Plugin entry point, lifecycle only
├── settings.ts       # Settings interface, defaults, and settings tab
├── commands/         # One file per command or command group
│   ├── index.ts      # Registers all commands on the plugin
│   └── my-command.ts
├── ui/               # Modals, views, and other UI components
│   ├── modal.ts
│   └── view.ts
├── utils/            # Pure utility functions and constants
│   ├── helpers.ts
│   └── constants.ts
└── types.ts          # Shared TypeScript interfaces and type aliases
```

This structure keeps each file under ~200–300 lines, makes it easy to find code by responsibility, and allows you to test utilities in isolation.
