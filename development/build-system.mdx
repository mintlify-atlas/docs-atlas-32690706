The build pipeline takes TypeScript source files in `src/` and produces a single `main.js` at the repo root that Obsidian loads as your plugin. [esbuild](https://esbuild.github.io) handles bundling — it is fast, zero-config for most cases, and produces the CommonJS output that Obsidian's plugin loader requires.

## npm scripts

Two scripts drive the entire build process:

<CodeGroup>

```bash npm run dev
npm run dev
```

```bash npm run build
npm run build
```

</CodeGroup>

| Script | What it does |
|--------|-------------|
| `npm run dev` | Starts esbuild in watch mode. Rebuilds `main.js` automatically whenever a source file changes. Includes inline sourcemaps for debugging. |
| `npm run build` | Runs `tsc -noEmit` first to type-check the project, then invokes esbuild in production mode: minified output, no sourcemaps. |

The `build` script runs `tsc -noEmit -skipLibCheck` before bundling. If TypeScript reports any type errors, the build stops before esbuild runs — so you can never ship a type-broken plugin by accident.

## esbuild config

The full build configuration lives in `esbuild.config.mjs`:

```javascript esbuild.config.mjs
import esbuild from "esbuild";
import process from "process";
import { builtinModules } from 'node:module';

const banner =
`/*
THIS IS A GENERATED/BUNDLED FILE BY ESBUILD
if you want to view the source, please visit the github repository of this plugin
*/
`;

const prod = (process.argv[2] === "production");

const context = await esbuild.context({
	banner: {
		js: banner,
	},
	entryPoints: ["src/main.ts"],
	bundle: true,
	external: [
		"obsidian",
		"electron",
		"@codemirror/autocomplete",
		"@codemirror/collab",
		"@codemirror/commands",
		"@codemirror/language",
		"@codemirror/lint",
		"@codemirror/search",
		"@codemirror/state",
		"@codemirror/view",
		"@lezer/common",
		"@lezer/highlight",
		"@lezer/lr",
		...builtinModules],
	format: "cjs",
	target: "es2018",
	logLevel: "info",
	sourcemap: prod ? false : "inline",
	treeShaking: true,
	outfile: "main.js",
	minify: prod,
});

if (prod) {
	await context.rebuild();
	process.exit(0);
} else {
	await context.watch();
}
```

## Key options explained

**`entryPoints: ["src/main.ts"]`**

esbuild starts from a single entry point. It follows all `import` statements recursively and bundles everything into one file. Your plugin only ever needs one entry point.

**`bundle: true`**

All imported modules are inlined into `main.js`. When Obsidian loads your plugin, it loads one file with no runtime dependencies to resolve.

**`external: [...]`**

Packages in this list are not bundled — they are left as `require(...)` calls. Obsidian injects these packages into the plugin's module scope at runtime:

- `"obsidian"` — the Obsidian API itself
- `"electron"` — Electron APIs (desktop only)
- All `@codemirror/*` and `@lezer/*` packages — the editor engine

Node.js built-in modules (`...builtinModules`) are also excluded; they are available in the desktop (Electron) environment but not on mobile.

**`format: "cjs"`**

Obsidian's plugin loader uses `require()` to load `main.js`, so the output must be CommonJS. Do not change this to `"esm"`.

**`target: "es2018"`**

esbuild transpiles down to ES2018 syntax. This provides broad compatibility across the range of Chromium versions that different Obsidian releases ship with.

**`outfile: "main.js"`**

The bundled output is written to the repo root as `main.js`. Obsidian expects `main.js` and `manifest.json` to be at the top level of the plugin folder.

**`treeShaking: true`**

Unused exports are removed from the bundle. If you import a utility library but only use one function, only that function ends up in `main.js`.

## Watch mode vs. production mode

The script checks `process.argv[2]` to determine which mode to use:

- **Dev** (`npm run dev`): Calls `context.watch()`. esbuild keeps a long-running process, watches the filesystem, and rebuilds incrementally on every change. Sourcemaps are inlined (`sourcemap: "inline"`), which lets the browser devtools show original TypeScript lines.

- **Production** (`npm run build`): Calls `context.rebuild()` once, then exits. `minify: true` removes whitespace and shortens identifiers. `sourcemap: false` keeps the bundle smaller and avoids shipping debug information.

## Build artifacts

`main.js` is a generated file. It must be present in the vault's plugin folder for Obsidian to load your plugin, but it should **never be committed to version control**. It is listed in `.gitignore` for this reason. Always regenerate it from source.

<Warning>
  Do not commit `main.js` to your repository. The file is generated from `src/` and should be reproduced by running `npm run build`. Committing it creates merge conflicts and obscures real code changes in pull request diffs.
</Warning>

Release artifacts (`main.js`, `manifest.json`, `styles.css`) are attached to GitHub releases as binary assets — not checked into source. See [GitHub release](/releasing/github-release) for the release workflow.
